# Hook Swing Web版 設計書

## 1. 構成
- `index.html`
  - タイトル/セレクト/ゲームの3画面DOM。
  - HUDとCanvasを配置。
- `style.css`
  - ダーク系ミニマルUI。
  - モバイル向けタップ前提スタイル。
- `script.js`
  - 状態管理、物理更新、描画、入力、保存処理を担当。

## 2. データモデル

### 2.1 ランタイム状態 `state`
- `progress`: 永続データ。
- `stages`: 50ステージ定義配列。
- `currentStageIndex`: 現在ステージ。
- `running`: ゲームループ中フラグ。
- `player`: 位置、速度、半径、ロープ長。
- `hookedAnchor`: 接続中フック点。
- `cameraX`: 横スクロールオフセット。
- `elapsedSec`: 現在タイム。

### 2.2 永続データ `progress`
- `reachedMax`: 到達済み最大番号。
- `cleared`: クリア済み辞書（例: `{ "3": true }`）。
- `bestTimes`: ステージ別ベスト秒数。

### 2.3 ステージ定義
- `anchors`: フック点配列。
- `goalLineX`: クリア判定X座標。
- `failZoneY`: 失敗ゾーンY座標。
- `failMargin`: 失敗猶予。
- `trampolines`: バウンス床配列（将来編集拡張対象）。

## 3. 処理フロー

### 3.1 起動
1. localStorage読み込み。
2. 50ステージ生成。
3. UI初期化。
4. タイトル画面表示。

### 3.2 ステージ開始
1. `currentStageIndex`設定。
2. 到達済み更新（開始時点）。
3. プレイヤー初期化。
4. `running = true` でループ開始。

### 3.3 物理更新（固定ステップ）
1. 重力加算。
2. フック中ならロープ長制約と半径方向速度除去。
3. 速度で位置更新。
4. トランポリン判定。
5. 失敗/クリア判定。
6. カメラとタイマー更新。

### 3.4 失敗処理
1. 失敗SE再生。
2. 同ステージを即再開始。

### 3.5 クリア処理
1. クリア済み更新。
2. ベスト更新。
3. 次ステージ解放。
4. 保存後、セレクト画面へ。

## 4. 入力設計
- `pointerdown` on canvas: フック接続。
- `pointerup/pointercancel` on window: フック解除。
- `touch-action: none`で誤スクロールを低減。

## 5. 描画設計
- 背景グリッド（速度感補助）。
- アンカー、ゴールライン、失敗ゾーン。
- 接続中ロープ。
- スティックマン簡易描画。
- カメラはプレイヤー追従（Xのみ）。

## 6. 将来拡張設計（ステージ作成モード）
- ステージ定義をJSON化して読み書き可能にする。
- エディタモードで以下をドラッグ編集対象にする。
  - フック点
  - トランポリン
  - ゴールライン
  - 失敗ゾーン
- 同一物理コードで即テストプレイ可能にする。
